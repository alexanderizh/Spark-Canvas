# 调试模式状态机 —— 完整流程与边界话术

```
[1 收集]  读 bugDescription（+ 必要的 Read/Grep）
   │
[2 启动]  begin → sid / port / round / snippets
   │
[3 假设]  形成一个明确假设
   │
[4 插桩]  评估影响面 → 插入带 __SPARK_DEBUG 标记的上报日志
   │
[5 等待]  给复现指引 → 结束 turn ───────────────┐
   │                                           │（用户去复现，日志后台累积）
[6 确认]  用户「复现完了」                        │
   │                                           │
[7 看量]  status.thisRound == 0 ? ──是──→ 提示没走到/调整 → 回 [4]
   │ 否
[8 读取]  read 本轮日志，对照假设
   │
   ├─ 假设成立 ───────────────→ [9 修复]
   └─ 不成立/信息不足 → next_round(新假设) → 回 [4]
   │
[9 修复]  评估影响面 → 改代码 → 插「验证日志」→ next_round('验证修复')
   │
[10 复测] 给复现指引 → 结束 turn
   │
[11 验收] 用户反馈
   ├─ 「解决了」→ finish → grep 删插桩 → 再 grep 校验 0 残留 → 交付
   └─ 「没解决」→ read → 回 [8]
```

## 关键话术

**插桩后交回用户（结束 turn）**
> 我在 `<文件:行>` 加了几处调试日志，假设是 **<一句话假设>**。请按以下步骤复现：①… ②… ③…。复现后回我一句「复现完了」，我来读日志分析。

**本轮没收到日志（thisRound==0）**
> 这一轮没收到日志，说明复现时大概率没走到我插桩的 `<路径>`。可能是 ① 触发条件不同 ② 代码走了另一分支。我把日志点挪到 `<新位置>`，麻烦再复现一次。

**多轮未定位（达护栏）**
> 已经排查 N 轮，排除了：<假设1>、<假设2>…。目前最可能的方向是 <X>，但还缺关键信息。能否补充：①更精确的复现条件 ②运行环境/版本 ③相关的报错或截图？这样我能更快定位。

**交付**
> 已解决。根因：<根因>；修复：`<文件:行>` 的 <改动>；证据：第 N 轮日志显示 <关键证据>。所有调试插桩已清除（grep `__SPARK_DEBUG` 零残留）。

## 反模式（不要做）

- ❌ 说"我已复现该问题" —— 你不能复现，复现是用户的事。
- ❌ 一轮插一大堆不相关日志 —— 难归因，按假设精准插。
- ❌ 对空日志强行脑补结论 —— 先确认 `thisRound > 0`。
- ❌ 重复一个已被 `status.hypotheses` 记录、已排除的假设。
- ❌ 交付时残留 `__SPARK_DEBUG` 代码 —— 必须 grep 校验为 0。
- ❌ 修改上报器的 `try/catch` / `keepalive` 让它能抛错 —— 会污染宿主应用。
