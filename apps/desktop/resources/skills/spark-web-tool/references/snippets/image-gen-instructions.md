## AI 图片生成能力（积极使用）

你拥有 AI 图片生成能力，配额为 **{{MAX_IMAGE_COUNT}} 张**。

### 核心使用原则

- **封面图必选**：标题页/封面必须生成一张配图
- **用满配额（重要）**：配额 {{MAX_IMAGE_COUNT}} 张是为本任务专门分配的视觉预算，**默认就应该尽量用满**。目标是用到配额的 90%~100%，而不是浅尝辄止只用两三张。图片能显著提升产物质感，宁可多配图也不要让配额闲置。
- **主动找配图位**：不要被动等"非配不可"的场景才生图。请主动审视每一页/每一节，思考"这里加一张图会不会更好"——封面、每个主要章节的概念插图、关键知识点、流程步骤、案例场景、总结页都应优先考虑配图，把剩余配额分配到这些位置，直到接近用满。
- **配额没用完要自查**：如果规划下来用到的图片数明显少于 {{MAX_IMAGE_COUNT}}（比如只用了不到一半），说明你很可能漏掉了可以配图的位置，必须重新审视产物、补充配图，而不是就此收手。
- **硬限保障**：接口在达到 {{MAX_IMAGE_COUNT}} 次后自动拒绝（HTTP 429）。收到 429 表示配额已用尽，**必须立即停止再发起任何生图请求**，对剩余位置改用「失败兜底」里的替代视觉元素，不要重试、不要换其它方式绕过

### 图片使用场景分类（按优先级）

| 图片类型 | 使用位置 | 优先级 | 推荐尺寸 |
|---------|---------|--------|---------|
| 封面图 | 标题页/封面 | 必须 | 1728x960（横版） |
| 背景装饰图 | 内容页背景（加遮罩） | 高 | 1728x960 |
| 概念插图 | 关键知识点讲解页 | 高 | 1280x1280 或 1568x1056 |
| 流程/信息图 | 流程图/步骤说明 | 中 | 1568x1056 |
| 海报/总结图 | 总结页/结尾页 | 中 | 1728x960 或 1280x1280 |
| 场景/产品图 | 案例/例题页 | 低 | 1568x1056 |

### 图片规划（生成前必须执行）

在调用图片生成接口前，先根据场景/内容规划图片分配：

1. 列出所有适合配图的场景（尽量挖掘，不要只列出"必须"的，凡是加图能更好表达的都列上）
2. 按优先级排序：封面 > 背景图/概念插图 > 流程图/海报 > 场景图
3. **尽量把配额排满**：从高优先级往下分配，直到接近用满 {{MAX_IMAGE_COUNT}} 张；只有当确实再也找不到合适配图位时才停止，而不是用够"最低限度"就停
4. 为每张图片确定：类型、英文 prompt 描述（30-80词）、推荐尺寸、语义化文件名

**判断标准**：如果图片能帮助受众更好地理解内容（如概念可视化、场景还原、流程示意、氛围烘托），就应生成。纯文字推导/公式计算等内容如果图片无助于理解，可以不配图。

### 调用方式（唯一通道：HTTP 接口）

```bash
curl -s -X POST http://localhost:{{PORT}}/api/v1/internal/image-gen \
  -H "Authorization: Bearer $IMAGE_GEN_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"task_id": {{TASK_ID}}, "prompt": "图片描述", "size": "1024x1024", "output_dir": {{OUTPUT_DIR_JSON}}, "defer": true, "filename": "cover_image.png"}'
```

**只能使用上面这个 HTTP 接口生图**——它统一管理配额计数。不存在其它生图脚本或命令；如果接口返回 429（配额用尽）或报错，按「失败兜底」处理，不要尝试任何其它方式生图。

### 推荐工作流（重要）

为避免图片生成阻塞整个产物流程，默认采用"先占位、后回填"的方式：

1. **尽早发起** HTTP 请求，并设置 `"defer": true`
2. 请求返回后会立即得到一个最终图片路径（如 `.../cover_image.png`），该路径已经创建了占位图文件
3. 你可以立刻在 HTML / build.mjs / 其他产物中引用这个路径，继续后续内容生成
4. **在最终导出/构建前必须等待图片收敛**：

```bash
curl -s -X POST http://localhost:{{PORT}}/api/v1/internal/image-gen/wait \
  -H "Authorization: Bearer $IMAGE_GEN_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"task_id": {{TASK_ID}}, "timeout_ms": 600000}'
```

5. 若等待成功，最终图片会覆盖原占位图，路径保持不变
6. 若等待超时或失败，保留占位图继续完成产物，不得因此中断

### 文件路径规则（必须遵守）

- 当前任务图片目录固定为：`{{OUTPUT_DIR_SHELL}}`
- 必须把 `output_dir` 传为上面的绝对路径，不要再传 `output/` 这样的相对路径
- 若想让最终文件名稳定可预期，请显式传 `filename`，例如 `cover_image.png`、`summary_image.png`
- 每次成功请求返回的 `filename` 都必须被写入最终产物中；不要只插入封面而遗漏其他已生成图片
- 多张图片请使用不同且语义明确的文件名，例如 `cover_image.png`、`concept_force_diagram.png`、`summary_map.png`
- 生成多张图片后，请在最终检查时搜索产物源码，确认每个生成图片文件名至少出现 1 次

### 调用限制

- 每个任务**最多调用 {{MAX_IMAGE_COUNT}} 次**，接口达上限后返回 HTTP 429
- 接口返回 `success: true` 时，`file_path` 为图片本地路径，`filename` 为文件名
- 当 `defer: true` 时，返回的 `file_path` 先对应占位图，后台完成后会被真实图片原地覆盖

### 图片尺寸要求

- 长宽范围：512px ~ 2048px，且必须为 32 的整数倍
- 推荐尺寸：1280x1280、1568x1056、1056x1568、1472x1088、1088x1472、1728x960、960x1728
- 不传 size 参数时默认 1280x1280
- 横版用 1728x960 或 1568x1056，竖版用 960x1728 或 1056x1568，正方形用 1280x1280

### 使用规则

1. 图片保存在当前任务的临时 workspace 目录中，不要写到项目根目录
2. HTML 产物中优先通过相对路径引用，例如文件位于同一目录时可用：`<img src="cover_image.png">`
3. PPTX 产物中在 build.mjs 里读取本地文件并用 `slide.addImage({ path: 'cover_image.png' })` 嵌入；**执行 build.mjs 前先调用 wait 接口**
4. 封面图、背景图、概念插图、海报图、场景图均可应用，按内容需要选择
5. 若生成了多张图片，必须把它们分别插入最相关的页面/章节；无法使用时不要发起对应图片请求

### 失败兜底（必须遵守）

如果图片生成失败（接口报错、超时、返回异常），**绝对不能因此中断产物生成**。必须使用以下替代方案之一：

- 使用文字描述或标题替代（在图片位置放置带样式的文字块或渐变色块）
- 使用 SVG 矢量图形替代（简单插图、图标、几何图形）
- 使用纯色/渐变背景 + 图标占位
- 使用免费图库图片作为后备（如 Unsplash）

**无论图片生成是否成功，产物必须完整生成。**

### 最终检查（生成完成后必须执行）

1. 搜索产物源码，确认每个成功生成的图片文件名至少出现 1 次
2. 确认所有图片都被嵌入到了最相关的页面/章节中
3. 确认没有遗漏已生成但未引用的图片
