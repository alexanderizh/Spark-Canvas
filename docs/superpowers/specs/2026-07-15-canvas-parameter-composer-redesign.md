# 画布参数配置与菜单体验重设计

> 状态: 已落地 | 最后核对: 2026-07-15

## 目标

重设计画布 AI 配置卡的模型与参数选择体验，使模型具备清晰的 Provider 层级和品牌图标，使比例、分辨率、数量等高频参数可视化，并将高频参数摘要与图标操作统一收拢到底部工具栏。同时整理右键菜单中的故事板与 360 全景图入口，并修复画布项目卡片“打开”按钮无响应的问题。

## 范围

本次改造覆盖：

- `CanvasInlineAiComposer`、`CanvasOperationPanel` 与 `CanvasOperationPresetModal` 的模型选择、Schema 参数展示、常用/高级参数组织和底部操作栏；所有画布任务配置入口共享同一套展示组件。
- 单节点右键菜单及相关节点快捷入口中的生成操作分组。
- `CanvasProjectsView` 项目卡片的显式打开交互。
- 与上述交互直接相关的样式、展示模型和自动化测试。

任务提交、参数裁剪、草稿恢复、模型能力过滤、Provider 调用、任务运行与产物落盘保持现有业务语义。

## 设计原则

1. 最佳体验优先，不以最小代码改动为约束。
2. 展示逻辑模块化，避免继续扩大当前约 2000 行的 `CanvasInlineAiComposer.tsx`。
3. Schema 驱动，但不把纯 UI 展示元数据耦合进底层媒体调用协议。
4. 已识别参数使用专用控件，未知参数可靠降级为通用输入控件。
5. 所有图标按钮具备 Tooltip、`aria-label`、键盘焦点和明确的禁用状态。
6. 不重复定义菜单分类，右键菜单和相关快捷入口共享同一展示模型。

## 模型选择器

模型选择器使用自定义双栏弹层：

- 触发器展示当前 Provider 图标、模型名和必要的能力摘要；任务面板未选择时明确提示“未选择模型”，允许不固定模型的预设入口提供“沿用平台默认”选项。
- 左栏按 `providerProfileId` 分组，显示 Provider 名称、品牌图标和可用模型数量。
- 右栏显示当前 Provider 下的模型，包含模型展示名、实际模型 ID、能力类型和选中状态。
- 同名模型在不同 Provider 下保持独立选项，稳定值继续使用 `providerProfileId + manifestId + effectiveModelId`。
- 搜索同时匹配 Provider 名称、Provider kind、模型展示名、manifest ID 和 effective model ID。
- 支持键盘导航、Escape 关闭、空状态、加载状态和超长列表滚动。
- 品牌图标复用现有 `ProviderLogo`、`getProviderIconForVendor` 与本地回退能力。

## 参数展示引擎

新增渲染端参数展示语义层，将 `SchemaField` 归一为以下控件：

- 比例：带真实宽高趋势的缩略框宫格，覆盖 `aspectRatio`、`aspect_ratio` 及作为比例使用的 `size`。
- 分辨率：紧凑分段按钮或可横向滚动的选项组。
- 生成数量：带数量单位的分段按钮。
- 时长：带秒数单位的分段按钮。
- 布尔值：开关。
- 普通枚举：下拉选择器。
- 允许自定义的枚举：可搜索、可输入的组合选择器。
- 数字和文本：保留受控输入框。

别名识别、展示类型、单位、图标和常用级别由 renderer 内的纯函数注册表负责。新模型 Schema 中未识别的字段进入通用控件并默认归入高级设置，不会丢失配置能力。

### 常用参数规则

- 所有媒体操作：模型。
- 图像：比例、分辨率/尺寸、生成数量、质量。
- 视频：比例、分辨率、时长、帧率、是否生成音频。
- 音频：音色、格式、时长或采样相关的高频字段。
- 文本：专属 Agent 与当前已有的主要生成选项。

输入传输、项目提示词注入、自定义私有参数、seed、服务档位、水印、搜索开关及无法识别的 Schema 字段默认进入高级设置。操作或模型自身只有少量字段时，分类仍由实际字段决定，不渲染空分组。

## 配置卡布局

配置卡保留主提示词编辑区域，底部改为统一工具栏；节点内联面板、展开面板与节点预设中心共享相同的模型/参数控件：

- 左侧依次显示模型、比例、分辨率、数量或时长等摘要按钮。
- 点击摘要按钮直接打开对应选择器，不要求用户进入完整高级面板。
- 右侧显示高级设置开关、辅助操作、现有消耗提示以及提交按钮。
- 取消、关闭、提交和其他底部动作改为图标按钮，文字通过 Tooltip 和无障碍标签提供。
- 高级设置默认折叠，展开状态写入独立的 renderer 本地偏好键并在会话间恢复。
- 窄宽度下参数摘要允许水平滚动，右侧主操作保持可见；极窄宽度下再换为两行布局。
- 全屏模式、浮动模式与预设配置共享同一组件和交互，不复制模型/参数实现。

## 菜单整理

新增统一的节点生成菜单展示模型：

- “剧本流水线”子菜单包含现有流水线动作以及“故事板”“360 全景图”。
- “生成任务”子菜单不再重复展示故事板和 360 全景图，只保留通用图像、文本、视频和音频任务。
- 右键菜单、浮动节点工具栏及其他复用同一操作目录的入口从统一展示模型读取，避免分类漂移。
- 360 全景产物节点的“全景预览”是产物查看动作，不属于创建动作，继续保留在菜单顶部。
- 底层创建操作仍分别使用 `storyboard_grid` 和 `panorama_360`，不改变任务类型。

## 画布项目打开修复

当前项目卡片操作区调用 `stopPropagation()`，而“打开”按钮没有自己的 `onClick`，因此点击按钮不会触发卡片级打开处理。

修复方式：

- 为“打开”按钮显式绑定 `handleOpenProject(project.id)`。
- 保留操作区事件隔离，避免菜单按钮误触发项目打开。
- 打开期间禁用当前项目的打开按钮并显示加载状态，阻止重复请求。
- 卡片整体点击仍可打开项目，按钮与卡片共用同一处理函数和错误反馈。

## 模块边界

- `CanvasInlineAiComposer.tsx`：保留状态编排、草稿恢复、任务 payload 组装和提交。
- `CanvasOperationPanel.tsx`：实际任务节点的内联/展开配置入口，复用统一模型与参数控件并保留原提交链路。
- `CanvasOperationPresetModal.tsx`：应用级节点预设入口，复用统一模型与参数控件并保留原预设存储链路。
- `canvasModelPickerModel.ts`：Provider 分组、搜索、稳定键和选中项解析的纯逻辑。
- `CanvasModelPicker.tsx`：双栏模型选择 UI。
- `CanvasOperationParameterControls.tsx`：工具栏与面板两种布局的统一模型/参数组合控件。
- `canvasParameterPresentation.ts`：Schema 字段语义识别、常用/高级分类、比例图形信息和摘要生成。
- `CanvasParameterControl.tsx`：按展示语义渲染具体参数控件。
- `CanvasComposerToolbar.tsx`：底部摘要、折叠控制和图标动作。
- `canvasNodeGenerationMenu.ts`：节点创建动作的统一分类模型。
- 现有 `ProviderLogo.tsx`：提供 Provider 品牌图标，不重复建设图标目录。

## 状态与数据流

1. Composer 根据 operation 筛选媒体模型。
2. 模型展示模型按 Provider 分组，选择结果仍回写现有 `selectedModelKey`。
3. 现有 capability、默认值和模型建议字段生成 `SchemaField[]`。
4. 参数展示语义层把字段划分为常用与高级，并为每个字段选择控件类型。
5. 所有控件继续写入现有 `modelParamDraft`，不引入第二份参数状态。
6. 提交时继续调用现有 `buildModelParams`、`normalizeModelParamsForSubmit` 和 `pruneModelParamsForCanvas`。

## 错误与边界处理

- 模型目录加载失败时保留自动路由并展示可恢复的空状态。
- Provider 无可识别图标时使用 `ProviderLogo` 的通用回退。
- 非法或未知比例值使用中性框形，不阻止选择和提交。
- 参数枚举数量过多时使用可滚动或搜索式选择器，避免撑破弹层。
- 高级参数为空时不渲染折叠区。
- 本地偏好读取失败时回退为默认折叠，不影响任务创建。
- 打开项目失败继续使用现有 warning/error 分级反馈，并恢复按钮状态。

## 测试策略

- 纯函数测试：Provider 分组、搜索、稳定键、参数语义识别、常用/高级分类、比例形状和摘要文本。
- 组件测试：双栏模型切换与选择、比例选择、折叠状态、图标按钮的 Tooltip/aria、项目打开按钮事件。
- 菜单测试：故事板和 360 全景图只出现在剧本流水线，通用生成菜单无重复项，全景预览仍保留。
- 回归测试：现有参数默认值、草稿恢复、提交 payload 和参数裁剪测试继续通过。
- 验证命令：相关 Vitest、desktop typecheck、desktop lint、desktop build；必要时通过本地 Electron/浏览器自动化核对布局与交互。

## 影响与风险

GitNexus MCP 当前未暴露，无法执行专属 impact/detect_changes。实施时按仓库降级规则，在每个生产符号修改前使用直接调用点检索确认影响范围，并在完成后使用相关测试和 `git diff` 核对变更范围。

高风险点主要是 Composer 草稿与提交链路。通过保持 `modelParamDraft`、模型稳定键和提交函数不变，将风险限制在展示层；菜单重构通过纯展示模型测试防止操作类型映射错误。
