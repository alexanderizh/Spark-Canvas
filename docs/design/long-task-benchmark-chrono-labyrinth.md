# 长任务基准：时间回廊网页游戏（任务文档 + 提示词）

> 状态: 待开发 | 最后核对: 2026-07-15

## 任务目标

在现有仓库中实现一个可直接运行的网页小游戏《时间回廊（Chrono Labyrinth）》，优先落在 `apps/website`，作为一个独立可访问的页面或路由。

这个任务的目的不是只测“能不能写个页面”，而是集中测试代理在长链路任务中的这些能力：

- 前期审查与方案拆解
- 多模块持续实现
- 中途保持上下文与约束一致
- 自主补测试与验证
- 收尾阶段的文档、验收和风险说明

## 为什么这个任务适合测超长执行

这个任务天然跨越产品设计、状态建模、交互实现、可玩性调优、存档恢复、可测试性和文档收尾，不容易被“一次性糊一个页面”糊弄过去。

相比做纯展示页，它更容易暴露长任务里的典型问题：

- 只会做 UI，不会做规则引擎
- 做到一半失去结构，代码开始堆在单文件
- 忘记补教程、失败态、重开、存档等闭环
- 没有测试 deterministic logic，后期改动容易自我打架
- 最后没有构建验证和文档落地

## 任务定义

### 游戏类型

做一个**回合制、确定性、种子驱动**的网格地牢小游戏。

玩家在一个未来感地牢中逐层前进，利用移动、攻击、技能和道具，在敌人行动前完成规划。整体体验应偏“策略 roguelike”，而不是纯实时动作游戏。

### 核心玩法要求

- 地图为网格制，建议 10x10 到 16x16。
- 每次行动后敌人统一按规则行动，保证回合逻辑清晰可追踪。
- 使用随机种子生成地图和事件；同一种子应产生相同关卡流程。
- 至少有 3 类普通敌人，行为模式要明显不同。
- 至少有 1 个精英/Boss 关卡。
- 至少有 4 种可获得的道具/能力升级。
- 有明确的胜利和失败条件。
- 支持重新开始，并展示本局关键结果。

### 产品级闭环要求

- 有开始界面、玩法说明、游戏中 HUD、结算界面。
- 有本地存档或断点续玩能力，至少支持“刷新页面后继续上一局”。
- 显示当前 seed，允许基于 seed 重开。
- 支持键盘操作；鼠标交互可选但建议补齐。
- 对窄屏布局做基本兼容，不能只在桌面大屏可用。
- 要有基础音效开关或动效降级开关，体现产品完成度。

## 建议功能规格

为避免任务太散，建议按下面这个规格做，不要求像商业游戏那样复杂，但要求闭环完整。

### 世界结构

- 3 个区域，每个区域 3 到 5 层。
- 每层包含：出生点、出口、战斗房、事件格、补给格中的若干种。
- 每通过一个区域，玩家获得一次更强升级或恢复机会。

### 玩家系统

- 生命值
- 能量/行动点
- 普通攻击
- 1 个位移或防御技能
- 1 个可成长的被动系统

### 敌人建议

- 追击型：只会逼近并近战攻击
- 射击型：保持距离，直线攻击
- 守卫型：低移动高护甲
- 召唤/干扰型：制造陷阱、护盾或额外单位

### 道具/升级建议

- 增加最大生命
- 增加行动点回复
- 攻击附带击退/穿透
- 技能冷却降低
- 首次受击护盾
- 击杀回复少量生命

### 可视反馈

- 玩家选中时高亮可移动范围或攻击目标
- 敌人攻击意图有提示
- 结算页展示：到达层数、击败敌人数、使用 seed、总用时

## 技术约束

- 优先使用现有仓库技术栈，不为了小游戏引入重型框架。
- 优先落在 `apps/website`，避免侵入桌面主流程。
- 代码保持模块化；如果单文件开始膨胀，及时拆到 `game/`、`engine/`、`content/`、`ui/` 等目录。
- 不依赖后端服务；排行榜、存档、统计都可以只做本地版本。
- 要遵守仓库里的文档和实现约束，包括 `docs/` 状态行规则。

## 验收标准

满足以下条件时，可以认为这个长任务“完成得不错”：

- 新增一个可访问的游戏页面，构建通过。
- 游戏具备完整开始 -> 游玩 -> 结算 -> 重开闭环。
- 同一 seed 下核心地图和敌人行为是稳定可复现的。
- 至少有一组针对核心逻辑的自动化测试。
- 页面不是 demo 级半成品，而是具备基本可玩性和说明。
- 对已有站点没有明显破坏。
- 最终有对应文档说明实现范围、玩法和验证方式。

## 评分观察点

你在测代理时，可以重点看这几项：

- 它是否先审仓库再动手，而不是盲写
- 它是否会主动拆阶段并持续推进
- 它是否会把规则引擎和 UI 分层
- 它是否会补 deterministic 测试
- 它是否会自己收尾：文档、构建、风险说明
- 它遇到中途复杂度上升时，是继续推进还是开始反复确认

## 推荐提示词

下面这版提示词适合直接丢给代理执行，重点是让它不要停在计划，而是端到端交付。

```text
请在这个仓库里实现一个“超长任务基准”功能，用来测试代理的长时间持续开发能力。

任务目标：
在现有项目中新增一个可玩的网页小游戏《时间回廊（Chrono Labyrinth）》。

整体要求：
1. 优先在 `apps/website` 中实现，做成一个独立可访问的页面或路由。
2. 游戏是回合制、网格制、确定性、seed 驱动的轻量 roguelike。
3. 需要完整闭环：开始界面、玩法说明、正式游玩、胜负结算、重新开始。
4. 需要本地存档或断点续玩能力，至少做到刷新后能继续上一局。
5. 需要展示当前 seed，并支持基于 seed 重开，保证同一 seed 下核心流程可复现。
6. 至少实现 3 类普通敌人、1 个精英或 Boss、4 种道具/成长项。
7. 需要有基础 HUD、操作提示、失败态和胜利态。
8. 需要兼容基本窄屏场景，不能只在桌面宽屏下可用。
9. 需要为核心规则补自动化测试，重点覆盖 deterministic logic、状态流转、存档序列化。
10. 最终要补对应文档，遵守仓库 `docs/` 状态行规则。

执行方式要求：
1. 先审查仓库结构、现有 website 路由、样式组织方式和测试方式，再动手实现。
2. 遵守仓库内 AGENTS.md 和相关约束；如果 GitNexus 当前健康且这次改动涉及复杂调用关系，可按仓库规则使用；如果不适合，就直接读源码，不要把 GitNexus 当成阻塞项。
3. 你要自主推进，不要只给计划；默认直接完成代码、测试、文档和验证。
4. 尽量复用现有依赖和风格，避免引入重量级新库；如果必须加依赖，要说明理由。
5. 保持模块化，不要把规则、内容、组件、存档逻辑全堆进一个文件。
6. 每完成一个主要阶段都继续往下做，直到端到端可交付。
7. 如果遇到不确定的小设计，请自行做合理决策并继续，不要频繁停下来问。

建议里程碑：
1. 仓库审查与方案落点
2. 游戏状态模型与 seed/回合引擎
3. 页面与交互实现
4. 存档、结算、教程、可视反馈
5. 测试、构建验证、文档补齐

最低验收标准：
1. 能运行并可玩
2. 构建通过
3. 有核心自动化测试
4. 有文档
5. 不破坏现有站点

完成后请输出：
1. 你新增/修改了什么
2. 运行了哪些验证命令，结果如何
3. 还剩哪些已知限制或后续可优化点
```

## 更狠一点的加压版

如果你想把测试难度再拉高，可以在提示词末尾再加这一段：

```text
额外要求：
1. 不要停在“能玩”层面，请尽量把玩法做出一点策略深度和反馈质感。
2. 为关键逻辑建立清晰模块边界，让后续继续扩展敌人、道具、地图规则时不会推倒重来。
3. 在实现过程中主动做阶段性自检，避免最后才发现路由、构建、测试或状态恢复有问题。
4. 如果某个方向复杂度过高，请主动降级设计，但必须保住完整闭环和可测性。
```

## 使用建议

- 如果你想测“能不能连续做 1 到 2 小时不跑偏”，用上面的推荐提示词就够。
- 如果你想测“遇到复杂状态管理会不会崩”，保留 seed、存档、测试这几个硬要求。
- 如果你想测“会不会只做表面 UI”，验收时重点看规则引擎、测试和结算闭环。
